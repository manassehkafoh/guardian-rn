# Lab 2 – Add a New Threat

> **Goal:** add a new threat type called `timeSpoofing` end-to-end — from the JSON schema through codegen, Kotlin, Swift, and the JS callback — and verify it appears in the example app.

This lab exercises the single-source-of-truth codegen pipeline that replaces the freerasp-rn three-way manual coupling. You will touch five files total; codegen handles the rest.

---

## Before you start

- Lab 1 must be complete and your Metro bundler must be running.
- You need a text editor open on the repository root.

---

## Step 1 — Add the threat to the JSON schema

Open `packages/schema/threat-schema.json`. Find the `threatId` enum array and append the new value:

```json
"threatId": {
  "type": "string",
  "enum": [
    "root",
    "jailbreak",
    "debugger",
    "hooks",
    "tamper",
    "emulator",
    "malware",
    "screenCapture",
    "timeSpoofing"
  ]
}
```

Save the file. Then validate the schema:

```bash
npm run schema:validate
```

Expected output:

```
✓ threat-schema.json is valid
```

---

## Step 2 — Run codegen

```bash
npm run codegen
```

Expected output — notice the new artefacts:

```
[codegen] ✓ TypeScript types    → packages/guardian-rn/src/generated/
[codegen] ✓ Kotlin sealed class → packages/guardian-rn/android/src/generated/
[codegen] ✓ Swift enum          → packages/guardian-rn/ios/Generated/
[codegen]   added: TimeSpoofing (all three targets)
```

Open `packages/guardian-rn/src/generated/ThreatId.ts` and confirm:

```typescript
export type ThreatId =
  | 'root' | 'jailbreak' | 'debugger' | 'hooks' | 'tamper'
  | 'emulator' | 'malware' | 'screenCapture' | 'timeSpoofing';
```

---

## Step 3 — Wire the Kotlin native side

Open `packages/guardian-rn/android/src/main/kotlin/com/guardian/rn/engine/community/CommunityEngine.kt`.

In the `PluginListener` inner class, add a case for the new threat inside the `onThreat` override:

```kotlin
ThreatType.TIME_SPOOFING -> emit(ThreatEvent.TimeSpoofing(evidence = buildEvidence()))
```

The `ThreatEvent.TimeSpoofing` data object was generated by codegen in Step 2 — you don't need to write it.

Run the Android unit tests to confirm nothing is broken:

```bash
./gradlew :guardian-rn:testDebugUnitTest
```

Expected output:

```
BUILD SUCCESSFUL in Xs
N tests completed, 0 failed
```

---

## Step 4 — Wire the Swift native side

Open `packages/guardian-rn/ios/Sources/GuardianRN/Engine/Community/CommunityEngine.swift`.

Inside the `SecurityThreatCenter` extension, add the new case to the `handle(_ threat:)` switch:

```swift
case .timeSpoofing:
    emitter.emit(ThreatEvent.timeSpoofing(evidence: buildEvidence()))
```

Run the iOS tests:

```bash
xcodebuild test \
  -workspace apps/example/ios/GuardianExample.xcworkspace \
  -scheme GuardianRN-Tests \
  -destination 'platform=iOS Simulator,name=iPhone 15'
```

Expected output:

```
Test Suite 'All tests' passed at ...
Executed N tests, with 0 failures
```

---

## Step 5 — Add the threat to the example app's callback list

Open `apps/example/src/App.tsx`. Add a handler for `timeSpoofing` inside the `useGuardian` call:

```typescript
timeSpoofing: () => {
  console.log('[guardian] threat: timeSpoofing');
  addThreat({ id: 'timeSpoofing', severity: 'medium', ts: Date.now() });
},
```

Save the file. Metro will hot-reload automatically.

---

## Step 6 — Write the TDD test first

Before adding the simulation button, write the test that should go red, then green.

Create `apps/example/src/__tests__/timeSpoofing.test.ts`:

```typescript
import { renderHook, act } from '@testing-library/react-hooks';
import { useGuardian } from '@guardian/rn';
import { mockEngine } from '@guardian/testing';

test('timeSpoofing callback is invoked when threat fires', async () => {
  const onTimeSpoofing = jest.fn();
  const { result } = renderHook(() =>
    useGuardian({ engine: mockEngine(), actions: { timeSpoofing: onTimeSpoofing } })
  );

  await act(async () => {
    mockEngine().emit('timeSpoofing', { confidence: 0.9 });
  });

  expect(onTimeSpoofing).toHaveBeenCalledTimes(1);
});
```

Run the test — it should **fail** (red):

```bash
npm test --workspace=apps/example -- --testPathPattern=timeSpoofing
```

Expected:

```
FAIL  src/__tests__/timeSpoofing.test.ts
  ● timeSpoofing callback is invoked when threat fires
    expect(jest.fn()).toHaveBeenCalledTimes(1)
    Received: 0
```

---

## Step 7 — Make the test pass (green)

The handler you added in Step 5 is all the production code needed. The test now uses the real `useGuardian` hook wired through the mock engine.

Re-run the test:

```bash
npm test --workspace=apps/example -- --testPathPattern=timeSpoofing
```

Expected:

```
PASS  src/__tests__/timeSpoofing.test.ts
  ✓ timeSpoofing callback is invoked when threat fires (42 ms)
```

## Step 8 — Trigger the threat manually

Add a **"Simulate timeSpoofing"** button to `apps/example/src/App.tsx`:

```typescript
<Button
  title="Simulate timeSpoofing"
  onPress={() => DevGuardian.simulateThreat('timeSpoofing')}
/>
```

`DevGuardian.simulateThreat` is a test-only helper that injects a synthetic event through the normal HMAC pipeline — only available in debug builds.

Tap the button in the running simulator. In the Metro console:

```
[guardian] threat: timeSpoofing
[guardian] policy: telemetry — event forwarded
```

The example UI shows a yellow threat card for `timeSpoofing`.

## Step 9 — Commit

```bash
git add packages/schema/threat-schema.json \
        packages/guardian-rn/android/src/main/kotlin/com/guardian/rn/engine/community/CommunityEngine.kt \
        packages/guardian-rn/ios/Sources/GuardianRN/Engine/Community/CommunityEngine.swift \
        apps/example/src/App.tsx \
        apps/example/src/__tests__/timeSpoofing.test.ts
git commit -m "feat(schema): add timeSpoofing threat type end-to-end"
```

CI will run codegen as a check step. If the generated files were not committed, the diff check fails and the PR is blocked.

---

## You should now understand

- The schema is the single source of truth: one change to `threat-schema.json` drives all three native targets and the TypeScript layer.
- Adding a threat touches exactly five files: schema, Kotlin engine, Swift engine, JS callback map, and the test. No switch statements to duplicate.
- The TDD red-green cycle proves the wiring before you touch the simulator.
- Codegen artefacts must be committed — the CI diff check enforces this.
