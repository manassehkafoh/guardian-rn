# ADR-0004: Engine interface, lifecycle, and conflation rules

## Status
Accepted (2026-05-10).

## Context
ADR-0001 committed to pluggable engines behind an `Engine` interface. This ADR specifies the exact interface contract, the lifecycle state machine, how multiple engines interact via the event bus, and the conflation rules that prevent a single real-event burst from generating an unbounded sequence of identical JS callbacks.

Key design tensions:
1. **Isolation:** an engine crash must not crash the TurboModule or other engines.
2. **Ordering:** the JS layer must not observe out-of-order sequence numbers across engines.
3. **Backpressure:** a misbehaving engine that emits 10,000 events/second must not flood the JS bridge.
4. **Health observability:** silent engine death (no events, no error) is the hardest failure mode to detect.

## Decision

### Interface (canonical TypeScript representation)

```typescript
interface Engine {
  /** Stable identifier for observability, e.g. "community@1.0.0" */
  readonly id: string;

  /**
   * Start detection. Must resolve before any event is emitted.
   * Throws EngineStartError if the engine cannot initialise.
   */
  start(context: EngineContext): Promise<void>;

  /**
   * Stop detection. Must be idempotent (safe to call twice).
   * Resolves when all in-flight events have been flushed to the bus.
   */
  stop(): Promise<void>;

  /** Stream of detected threat events. */
  readonly onThreat: Observable<ThreatEvent>;

  /**
   * Heartbeat emitted at least once every ENGINE_HEALTH_TICK_MAX_MS (60 000).
   * Absence triggers alert A-2.
   */
  readonly onHealthTick: Observable<EngineHealthTick>;
}

interface EngineContext {
  config: GuardianConfig;        // tenant-provided configuration
  sessionId: string;             // current session UUID
  platform: 'android' | 'ios';
}

interface EngineHealthTick {
  engineId: string;
  ts: number;        // Unix epoch ms
  activeChecks?: string[];  // optional: which detectors are running
}
```

The Kotlin and Swift equivalents are generated by codegen from this interface definition. The `Observable<T>` type maps to `Flow<T>` in Kotlin and `AnyPublisher<T, Never>` in Swift.

### Lifecycle state machine

```
  ┌──────────┐   start()    ┌─────────┐   stop()    ┌──────────┐
  │ CREATED  │ ──────────► │ RUNNING │ ──────────► │ STOPPED  │
  └──────────┘             └─────────┘             └──────────┘
                                │                        │
                           error thrown              stop() (noop)
                                │
                           ┌─────────┐
                           │ FAULTED │
                           └─────────┘
```

- **CREATED → RUNNING:** `start()` resolves. No events may be emitted before this transition.
- **RUNNING → STOPPED:** `stop()` resolves. Events emitted after `stop()` resolves are silently dropped by the bus. `stop()` from STOPPED is a no-op.
- **RUNNING → FAULTED:** `start()` throws, or the engine calls the internal `context.onFault(error)` callback for unrecoverable errors detected after start. The bus removes the engine from the active set, logs the fault, and emits a `guardian.engine.fault` observability event. The remaining engines continue unaffected.
- **FAULTED:** terminal. The guardian-rn bus will not call `start()` again without a full `NativeGuardianRN.restart()`.

### Multiple engines and the event bus

When `useGuardian` is called with an `engines` array:
1. The bus calls `start()` on each engine in declaration order.
2. All `onThreat` streams are merged into a single ordered stream using the HMAC sequence counter (not wall-clock time).
3. All `onHealthTick` streams are merged and forwarded to the telemetry adapter.
4. If one engine faults, the bus continues running the others and emits a synthetic `engineFault` event to the JS layer so the caller can observe the degradation.

### Conflation rules

To address backpressure and duplicate-event noise, the bus applies two conflation rules:

**Rule 1 — Deduplication window (100 ms):**
If the same `threatId` is emitted more than once within a 100 ms sliding window from the same `engineId`, only the first emission is forwarded to the JS callback. Subsequent duplicates within the window are counted and appended as `evidence.suppressedCount` to the forwarded event. This handles the common case of an engine that polls at a high rate and emits the same threat on each poll tick.

**Rule 2 — Per-engine rate cap (50 events/second):**
If a single engine emits more than 50 events/second (averaged over 1 second), the bus enters **throttle mode** for that engine: it drops events beyond the cap, increments `guardian.bus.droppedCount`, and emits a `busThrottle` system event. Throttle mode releases automatically when the rate drops below 25 events/second for 5 consecutive seconds.

Both thresholds are configurable via `GuardianConfig.busConfig` but the defaults are fixed in the schema.

### Health tick contract

- `ENGINE_HEALTH_TICK_MAX_MS = 60_000` — an engine that has not emitted a health tick within this window is considered silent.
- The bus tracks the last tick timestamp per engine. If any engine exceeds the window, it emits a `guardianHealthMiss` event and the telemetry adapter forwards it as `guardian.engine.healthMiss` in the ECS document, which triggers Grafana alert A-2.
- This replaces polling the engine with a query; the engine is responsible for emitting ticks even when no threats are detected.

## Consequences
- The fault-isolation model means a commercial engine that crashes cannot take down the community engine or the JS bridge. This is the primary benefit of the pluggable architecture.
- The deduplication window (100 ms) may suppress legitimate rapid-succession events from different physical causes that happen to share a `threatId`. Detection engineers must be aware of this when tuning engines that can legitimately fire the same threat in rapid succession.
- The rate cap (50 events/second) is generous for real threat scenarios but may trip during aggressive simulation testing. Test harnesses should use `DevGuardian.simulateThreat` which bypasses the bus rate cap.
- The requirement that `stop()` flush in-flight events before resolving adds latency to app shutdown. The bus sets a 2-second stop timeout; events not flushed within that window are dropped and logged.

## Alternatives considered
- **Push model via callbacks instead of Observable/Flow/Publisher** — rejected; Kotlin Flow and Swift Combine are now idiomatic for async streams on their respective platforms, and the structured-concurrency backpressure semantics are essential for the rate cap.
- **Single engine only (no pluggability)** — rejected per ADR-0001; recreates vendor lock-in.
- **Engine restart on fault** — rejected; a faulty engine may enter a crash loop, making the restart policy harder to reason about than a terminal FAULTED state. The operator restarts via `NativeGuardianRN.restart()` at their discretion.
- **No conflation, let consumers filter** — rejected; flooding the JS bridge with 10,000 events/second would degrade app performance regardless of whether the JS callback does anything with them.
