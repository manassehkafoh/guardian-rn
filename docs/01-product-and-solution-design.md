# freeRASP for React Native — Product & Solution Design Document

> **Source artifact analysed:** `~/Downloads/RASPReactNative-master` (npm package `freerasp-react-native@4.5.2`, Android SDK `TalsecSecurity-Community-ReactNative:18.0.4`, iOS `TalsecRuntime` xcframework 6.14.4).
> **Analysis date:** 2026-05-10.
> **Authors:** Reverse-engineered from source. Reflects the public open-source layer of the plugin only; the underlying Talsec native SDKs are closed-source binaries (`.aar` and `.xcframework`).

---

## 1. Executive Summary

`freerasp-react-native` is a thin React Native (RN) bridge that surfaces Talsec's commercial **freeRASP** (Runtime Application Self-Protection) detection engine to JS-land. The native engine — distributed as a closed-source AAR on Android and an XCFramework on iOS — runs ~22 security checks (root/jailbreak, hooks, debugger, emulator, tamper, untrusted store, malware, screen capture, etc.) and reports threats back to a **single React `useFreeRasp` hook** registered with a config + an actions map of per-threat callbacks.

The repo is roughly **4 layers**:

1. **TypeScript SDK** (`src/`) — public API, types, event-emitter listeners, channel + identifier negotiation, malware payload decoder.
2. **Android bridge** (`android/src/main/java/com/freeraspreactnative/*.kt`) — `ReactContextBaseJavaModule` wrapping `com.aheaditec.talsec_security` Talsec SDK, with a foreground-aware threat dispatcher, screenshot/screen-recording detector for Android 14+/15+, malware payload encoder, and an obfuscation-friendly random-int callback ID system.
3. **iOS bridge** (`ios/*.swift` + `.m`) — `RCTEventEmitter` wrapping `TalsecRuntime`, mirroring the Android dispatcher pattern, with a smaller threat surface (no malware, no screen-recording detection at the SDK layer).
4. **Expo config plugin** (`plugin/src/`) — auto-injects the Talsec maven repo and lets users pin `minSdkVersion`/R8 in `build.gradle`.

The package's central design idea is to push security state changes through a **single RN `NativeEventEmitter` channel**, but with **randomised, per-process channel/key/threat-ID strings generated by `SecureRandom`** so that an attacker statically analysing the bundle cannot pattern-match well-known constants like `"onRootDetected"`. This is its main innovation versus a naïve RN wrapper.

The product is **freemium**: the open-source layer in this repo is MIT-licensed; the underlying detection engine and "AppiCrypt" (commercial app-attestation cryptogram) live in the closed binaries and behind a paid licence. The free tier is gated by the `watcherMail` field, which is also how Talsec sends the weekly "Security Report" email.

The solution **works** but ships several material weaknesses (detailed in §6 and addressed in the companion document `02-superior-solution-proposal.md`): a singleton-coupled Kotlin module that conflates lifecycle with detection, an iOS bridge that silently swallows scene/window-acquisition failures, no New Architecture (Fabric/TurboModule) support, no JS-side type-safety on the `NativeEvent` payload, race conditions in `setThreatListeners`, and a security model that ultimately rests on `process.kill`/`abort()` — defences a sufficiently motivated attacker on a rooted device can defeat.

## 2. Product Context

### 2.1 What problem does it solve?

Mobile apps execute on **untrusted devices**. Unlike server-side software, the runtime is owned by the user (or attacker). Without RASP, the following threat classes are open:

| Threat class | Concrete attacks |
|---|---|
| Reverse engineering | Frida/Xposed hooks, Ghidra decompilation, IDA, dump-the-keys |
| Repackaging / tampering | Modified APK with re-signed certificate, in-app cheats, ad-injection |
| Compromised OS | Rooted Android, jailbroken iOS, Magisk Hide, Dopamine, unc0ver, KernelSU |
| Side-loading | Untrusted store installation, sideload via ADB |
| Dynamic instrumentation | Debugger attached, emulator/simulator, automation frameworks (Appium) |
| Data leakage | Screenshot, screen recording, screen mirroring |
| Identity / fraud | Multi-instance cloning, device-binding bypass, time/location spoofing, system VPN |
| Distribution malware | Malicious apps installed alongside legit one (Android only) |

freeRASP's value proposition: **detect** these conditions at runtime, hand the decision (kill app, degrade gracefully, telemetry-only) to the host application, and aggregate the data into Talsec's weekly e-mail report so security teams get visibility without rolling their own pipeline.

### 2.2 Who is it for?

- **Banking / fintech** apps (PSD2, FFIEC) where RASP is mandated or implied by regulator guidance.
- **Health** apps (HIPAA, GDPR special-category data) where on-device controls are required.
- **Crypto wallets, gaming, DRM-protected media, premium SaaS, public-sector apps**.
- **React Native shops** that don't want to write a Kotlin/Swift bridge to a commercial RASP themselves.

### 2.3 Pricing & licensing model (inferred)

- Open-source bridge: MIT.
- Native detection engine: closed-source freemium ("fair usage policy"). Free for low volume + telemetry to `watcherMail`.
- Commercial tier ("Business / Ultimate"): adds **AppiCrypt** (app-integrity cryptogram for backend API protection), priority support, no fair-usage cap, custom check thresholds.

### 2.4 Competitors (positioning frame)

| Vendor | Differentiator |
|---|---|
| Guardsquare DexGuard / iXGuard | App-shielding + RASP, JVM-deep obfuscation. Premium-only. |
| Promon SHIELD | Strong banking footprint, Self-Defending Apps. Premium-only. |
| Verimatrix XTD | Threat-feed style integration. Premium-only. |
| Approov | API-attestation focus, less RASP. Subscription. |
| **freeRASP (Talsec)** | **Free tier with real RASP**, plus paid AppiCrypt. The only meaningful free option in the space. |
| OWASP MASVS-only DIY | Free but reinventing the wheel; no malware list, no jailbreak signature feed. |

### 2.5 Success metrics

For the package itself (inferred from repo signals — GitHub stars, weekly email, 42matters Rising Star badge):

- **Adoption**: weekly NPM downloads, GitHub stars (visible in README badges).
- **Detection efficacy**: false-positive rate (CHANGELOG shows recurring iOS jailbreak FP fixes) vs. detection rate against new bypass tools (KernelSU, HMA support added in 4.5.0).
- **Latency**: README claims "low latency"; the `RaspExecutionState.AllChecksFinished` event is the only first-class signal a host app gets to know when synchronous checks are complete. No numeric SLO published.
- **Time-to-update against new bypass tools**: CHANGELOG cadence ≈ monthly minor releases.

### 3.1 Two parallel event channels

There are exactly **two channels** between native and JS:

1. **Threat channel** — emits one of 22 threat events (Android) or 14 (iOS).
2. **RaspExecutionState channel** — currently emits a single event, `AllChecksFinished`, signalling the synchronous detection batch is complete.

Both follow an identical 4-step contract:

| Step | Native API | JS API |
|---|---|---|
| 1. Pull channel descriptor | `getThreatChannelData()` | returns `[channelName, channelKey, malwareKey]` (random strings) |
| 2. Pull threat-ID dictionary | `getThreatIdentifiers()` | returns array of N random ints; index ↔ threat enum order |
| 3. Patch JS-side enum | `Threat.getValues()[i].value = newValues[i]` | mutates static class fields in `models/threat.ts` |
| 4. Subscribe | `eventEmitter.addListener(channelName, listener)` | listener switches on `event[channelKey]` |

This **bind-at-runtime** pattern is the package's most distinctive design choice. Static analysis of the JS bundle gives an attacker no useful constants; they have to dynamically tap the bridge, which is itself proxied through Talsec's closed-source code.

### 3.2 Lifecycle / threading

- **Android** uses `LifecycleEventListener` to flip an `isAppInForeground` boolean and replay cached threats on resume. `Talsec.start` is dispatched on the UI thread; promises resolve on the main `Handler`. App-icon Base64 encoding runs on a dedicated `HandlerThread` ("BackgroundThread") to avoid blocking JS.
- **iOS** caches threats while no listener is attached and flushes inside `didSet` on the dispatcher's `listener` property. Window/scene access is async-dispatched to the main queue (`getProtectedWindow`).
- **JS** uses module-level booleans (`isInitializing`, `isMappingPrepared`, `isRaspStarted`) to deduplicate `setThreatListeners` calls — these are **process-global**, so a second `useFreeRasp` invocation with different actions silently no-ops.

### 3.3 Data flow for a typical threat (root detection on Android)

1. Talsec native scheduler runs root checks during/after `Talsec.start()`.
2. `ThreatListener.ThreatDetected.onRootDetected()` fires inside the SDK.
3. Plugin's `PluginThreatHandler` translates → `ThreatDispatcher.dispatchThreat(ThreatEvent.PrivilegedAccess)`.
4. If foreground + listener registered: `PluginListener.threatDetected(event)` builds a `WritableMap` `{[CHANNEL_KEY]: event.value}` and emits via `RCTDeviceEventEmitter.emit(CHANNEL_NAME, params)`.
5. If background or no listener: `threatCache.add(event)` (synchronised on `threatCache`).
6. RN bridge delivers to the `eventEmitter` listener registered in `src/api/listeners/threat.ts`.
7. JS switches on `event[threatKey]`, matches against `Threat.PrivilegedAccess.value`, and calls `actions.privilegedAccess()`.

### 3.4 Malware payload (Android-only)

- SDK delivers `MutableList<SuspiciousAppInfo>` (Android `PackageInfo` + reason + permissions).
- Plugin maps each entry to its serializable `RNSuspiciousAppInfo` (kotlinx.serialization), JSON-encodes it, Base64-encodes the result, and pushes the array of strings under the random `MALWARE_CHANNEL_KEY`.
- JS `parseMalwareData` uses `Buffer.from(b64, 'base64')` (from the `buffer` polyfill) to decode each entry back to a typed `SuspiciousAppInfo`. App icons are deliberately **not** sent in this payload (heavy work) — `getAppIcon(packageName)` is a separate on-demand call.

### 3.5 Configuration shape

```ts
type TalsecConfig = {
  androidConfig?: { packageName, certificateHashes[], supportedAlternativeStores?, malwareConfig? },
  iosConfig?:     { appBundleId, appTeamId },
  watcherMail:  string,    // required — feeds Talsec weekly report
  isProd?:      boolean,   // default true; false → debug-only (less aggressive)
  killOnBypass?: boolean,  // Android-only: kill app if SDK integrity bypass detected
};
```

`malwareConfig` lets the host pin blacklisted hashes/package names, mark suspicious permission *combinations* (note the `string[][]`: each inner array is an AND group, the outer is OR), and whitelist installation sources beyond Play Store.

### 3.6 Packaging & build

- `react-native-builder-bob` produces three outputs (`commonjs`, `module`, `typescript`) under `lib/`.
- Android side ships **only Kotlin source**; the binary detection engine is pulled from `europe-west3-maven.pkg.dev/talsec-artifact-repository/freerasp` (Talsec's GAR). This means **anyone offline or behind a strict artifact proxy must mirror that repo first**.
- iOS ships the `TalsecRuntime.xcframework` directly inside `ios/`, vendored via `s.ios.vendored_frameworks` in the podspec — meaning the binary is in the npm tarball.
- Expo plugin auto-adds the maven repo + jitpack to the consumer's project `build.gradle`, with idempotency guarded by string-include checks (`buildGradle.includes(urlFreerasp)`).

## 4. Component-level Design

### 4.1 JS / TypeScript layer (`src/`)

| File | Purpose | Notes |
|---|---|---|
| `src/index.tsx` | Public surface. Re-exports types, methods, listeners. | No default export. |
| `src/types/types.ts` | All public types: `TalsecConfig`, `ThreatEventActions`, `SuspiciousAppInfo`, `RaspExecutionStateEventActions`, `NativeEvent`. | `NativeEvent` is loosely typed (`{[k]: number\|string[]\|undefined}`). |
| `src/api/methods/native.ts` | Thin `Promise`-returning wrappers around `NativeModules.FreeraspReactNative`. iOS-unsupported methods (`addToWhitelist`, `getAppIcon`) reject early. | |
| `src/api/methods/reactNative.ts` | The one and only `useFreeRasp` hook. | Module-level `isRaspStarted` makes it a singleton across the JS process. |
| `src/api/listeners/threat.ts` | Wires the threat channel, switch-cases on threat IDs, decodes malware payload. | 22-arm switch — every new threat requires editing this file. |
| `src/api/listeners/raspExecutionState.ts` | Same pattern, currently 1 event. | |
| `src/channels/threat.ts` | Bridge calls to fetch channel descriptor + identifiers, mutates `Threat.getValues()` in place. | |
| `src/channels/raspExecutionState.ts` | Same pattern. | |
| `src/models/threat.ts` | `Threat` class with 22 static instances. `getValues()` returns *different ordered arrays* per platform — index alignment with native is order-sensitive. | The `static AppIntegrity = new Threat(0)` etc. is a TS workaround for `enum` not allowing instance-mutation. |
| `src/models/raspExecutionState.ts` | Same pattern, single event. | |
| `src/utils/utils.ts` | `getThreatCount`, `getRaspExecutionStateCount`, `itemsHaveType` (loose `every typeof === t` check). | |
| `src/utils/malware.ts` | Base64+JSON decode of `SuspiciousAppInfo[]`. | Uses `buffer` package — not available natively in JS. |
| `src/internal.ts` | **Empty stub** retained for backwards compatibility. | Comment: `// Deprecated: logic moved under utils/*`. |

#### 4.1.1 The `Threat` model is order-sensitive (subtle)

In `Threat.getValues()`, Android returns a 21-element array, iOS returns 14 — and the **order must match exactly** the order in which the native side pushes IDs through `getThreatIdentifiers()`. Any reordering on either side desyncs every threat callback silently. The current code has no integrity check beyond the array-length comparison in `getThreatIdentifiers()`. This is a fragile contract.

#### 4.1.2 `useFreeRasp` race condition window

```ts
let isRaspStarted = false;          // (a) module-scope
useEffect(() => {
  (async () => {
    await setThreatListeners(actions);  // (b) async, may yield
    if (isRaspStarted) return;          // (c) check
    await talsecStart(config);          // (d) yield
    isRaspStarted = true;               // (e) set
  })();
}, []);
```

If two components mount concurrently (or React 18 strict-mode double-invokes the effect), both pass check (c) before (e) runs in either, leading to a duplicate `talsecStart`. The native module guards this on Android (`if (talsecStarted) return promise.resolve(...)`) but not on iOS — iOS will call `Talsec.start` twice. Whether that's safe is undocumented.

### 4.2 Android layer (`android/src/main/java/com/freeraspreactnative/`)

#### 4.2.1 `FreeraspReactNativeModule.kt` (357 LOC)

The single `ReactContextBaseJavaModule`. Responsibilities:

- `talsecStart(options, promise)` — parses the `ReadableMap` config (with `getStringThrowing`, `getMapThrowing`, `getArraySafe`, `getNestedArraySafe` extensions), builds a `TalsecConfig`, registers SDK listener, calls `Talsec.start` on the UI thread, marks `talsecStarted = true`, registers `ScreenProtector` if API 34+.
- Exposes 11 `@ReactMethod` functions: `addToWhitelist`, `blockScreenCapture`, `isScreenCaptureBlocked`, `getAppIcon`, `storeExternalId`, `removeExternalId`, `onInvalidCallback`, `addListener`, `removeListeners`, `removeListenerForEvent`, `getThreatIdentifiers`, `getThreatChannelData` (and the RASP-state equivalents).
- Owns a `LifecycleEventListener` that drives `ThreatDispatcher.onResume/onPause`, registers/unregisters `ScreenProtector`, and on `isFinishing` tears down all subscriptions.
- Embeds an inner `PluginListener` that implements both `PluginThreatListener` and `PluginExecutionStateListener` and emits via `RCTDeviceEventEmitter`.

#### 4.2.2 `PluginThreatHandler.kt`

Adapter from the **closed-source** `ThreatListener.ThreatDetected` / `ThreatListener.DeviceState` / `ThreatListener.RaspExecutionState` SDK callbacks to the plugin's internal `ThreatEvent` sealed class. There is one method per threat — adding a new threat requires touching this file *and* `ThreatEvent.kt` *and* `Threat.getValues()` in JS. Three coupled changes for one new check.

#### 4.2.3 `dispatchers/ThreatDispatcher.kt` & `ExecutionStateDispatcher.kt`

Singletons. They hold:

- `lateinit var listener` (set when JS attaches),
- two booleans `isAppInForeground` / `isListenerRegistered`,
- a `mutableSetOf<ThreatEvent>()` cache for missed threats.

The cache uses `Set` semantics — if Talsec fires `Hooks` 5 times in 5 ms while the app is backgrounded, JS gets one delivery on resume. This is reasonable for binary state ("hooked Y/N") but **loses count-of-events fidelity** for any future signal where frequency matters.

`flushCache()` is called from both `registerListener` and `onResume`, with a per-method snapshot under `synchronized(threatCache)`. The **listener invocation itself is outside the synchronized block** — that's correct (avoids holding the lock while crossing the bridge) but leaves a TOCTOU window where a fresh threat fires *during* flush. Such a threat lands back in `threatCache` if the foreground flag flipped — a subtle but acceptable behaviour.

#### 4.2.4 `events/ThreatEvent.kt` & `RaspExecutionStateEvent.kt`

Kotlin `sealed class` with `data object` instances. Each `data object`'s `value` is generated **once per process launch** via `RandomGenerator.next()` (`SecureRandom` over `[10000, 999_999_999]`, with set-deduplication). Channel name and key are likewise random integers stringified. Companion `ALL_EVENTS` is a `WritableArray` of all values in declared order.

**Why sealed class, not enum?** Comment in source: *"Sealed classes are used because of obfuscation - enums classes are not obfuscated well enough."* Combined with `proguard-rules.pro`'s `-repackageclasses 'ts'`, the compiled bytecode no longer reveals event names by class layout.

#### 4.2.5 `ScreenProtector.kt`

Android 14 (`UPSIDE_DOWN_CAKE`/API 34) introduced `Activity.registerScreenCaptureCallback`, and Android 15 (`VANILLA_ICE_CREAM`/API 35) added `WindowManager.addScreenRecordingCallback`. The plugin wraps both, with explicit `DETECT_SCREEN_CAPTURE`/`DETECT_SCREEN_RECORDING` permission checks and graceful logging when the host hasn't declared them.

Note: it **delegates the actual signalling** back to the SDK (`Talsec.onScreenshotDetected()` / `Talsec.onScreenRecordingDetected()`), which then routes through `PluginThreatHandler`. So the SDK is the single source of truth for "did a screenshot happen", but the OS-level callback is owned by the plugin because the SDK can't register lifecycle-bound callbacks itself.

#### 4.2.6 `utils/Extensions.kt`

A grab-bag of `ReadableMap`/`ReadableArray` ergonomics. Notable: `getNestedArraySafe` for `suspiciousPermissions: string[][]`, and `MutableList<SuspiciousAppInfo>.toEncodedWritableArray(context)` which JSON-serializes via `kotlinx.serialization` then Base64-encodes (`Base64.DEFAULT` — note the **default flavor includes line wrapping at 76 chars**, which is then decoded by JS's `Buffer.from(b64, 'base64')` — JS Buffer tolerates this, but it's fragile if a stricter decoder is ever swapped in).

#### 4.2.7 `RandomGenerator.kt`

A process-local `SecureRandom` with a `mutableSetOf<Int>` for de-dup. Range `[10000, 999_999_999]` ≈ 30 bits of entropy per ID, ~60 IDs total → collision probability is comfortably <2⁻¹⁵.

### 4.3 iOS layer (`ios/`)

#### 4.3.1 `FreeraspReactNative.swift` (181 LOC)

Subclass of `RCTEventEmitter`. Same shape as Android but more compact because the iOS Talsec SDK (`TalsecRuntime`) exposes a single `SecurityThreatHandler.threatDetected(_)` callback (plus `RaspExecutionState.onAllChecksFinished()`) — there are no per-threat methods to bridge.

The Swift extension `extension SecurityThreatCenter: @retroactive SecurityThreatHandler, @retroactive RaspExecutionState` is the pivot: it intercepts every threat coming out of the Talsec runtime and routes through `ThreatDispatcher.shared`. The `passcodeChange` raw-value is filtered out before dispatch (treated as an iOS-only diagnostic, not a security event).

`onInvalidCallback` calls `abort()`. Android calls `android.os.Process.killProcess(android.os.Process.myPid())`. Both are nuclear-option self-protections triggered when the JS receives an unknown threat ID — i.e. when an attacker has likely tampered with the bridge.

#### 4.3.2 `models/SecurityThreat.swift`

A **switch-on-rawValue** lookup that maps Talsec's enum cases to indices into `EventIdentifiers.threatIdentifierList[0..14]`. Identical positional contract to Android. `@unknown default → abort()`, again a nuclear bypass-detection.

#### 4.3.3 `dispatchers/`

`ThreatDispatcher` and `ExecutionStateDispatcher` are Swift `class` singletons with an `NSLock` — direct functional analogues of the Kotlin singletons but **without the foreground gate**. iOS relies purely on listener-attached vs not. The reason: iOS doesn't have the same `LifecycleEventListener` semantic; the plugin doesn't try to defer events while the app is in background, leaving that scheduling to RN itself.

#### 4.3.4 `utils/EventIdentifiers.swift`

Generates 20 random integers up-front via `RandomGenerator.generateRandomIdentifiers(length: 20)` and slices:

- `[0]` → threatChannelKey
- `[1]` → threatChannelName
- `[2]` → raspExecutionStateChannelKey
- `[3]` → raspExecutionStateChannelName
- `[4]` → AllChecksFinished identifier
- `.suffix(15)` → 15 threat identifiers

Note the **off-by-one risk**: only 14 iOS threats actually exist (filter on `passcodeChange` removes one), but `threatIdentifierList` is sized for 15 (`SecurityThreat.allCases.count` includes `passcodeChange`). This is intentional — the indices in `SecurityThreat.callbackIdentifier` line up with the unfiltered enum.

#### 4.3.5 `utils/RandomGenerator.swift`

Uses **Swift's standard `Int.random(in:)`**, NOT a CSPRNG. This is materially weaker than the Android side's `SecureRandom`. Given the threat model — making static analysis harder — this is *probably* sufficient (an attacker dynamically tapping the bridge wins regardless), but the asymmetry is worth flagging.

#### 4.3.6 `utils/Utils.swift`

`getProtectedWindow(completion:)` walks `UIApplication.shared.connectedScenes` to find the first `UIWindowScene` and its first `UIWindow`. **It silently passes `nil` to the completion** if anything's missing (e.g. multi-scene apps, or main-thread acquisition before scenes are connected). The caller (`blockScreenCapture`) does reject on nil — but `isScreenCaptureBlocked` likewise — there's no fallback to the legacy `UIApplication.shared.windows`. In a multi-scene iPad/CarPlay app, screen-capture blocking will silently target only the first scene.

### 4.4 Expo config plugin (`plugin/src/`)

- `withRnTalsecApp` composes three sub-plugins:
  1. `withBuildscriptDependency` — appends `allprojects { repositories { maven { url <Talsec GAR> } maven { url <Jitpack> } } }` to the root `build.gradle` if not already present.
  2. `withAndroidMinSdkVersion` — surfaces a `minSdkVersion` option via `createBuildGradlePropsConfigPlugin`.
  3. `withAndroidR8Version` — appends a `buildscript { dependencies { classpath("com.android.tools:r8:<v>") } }` block (used to pin a newer R8 if the user's RN baseline ships an older one, since Talsec's obfuscation pipeline benefits from current R8).
- Wrapped in `createRunOncePlugin(..., pkg.name, pkg.version)` for Expo idempotency.
- **Scope is Android-only** — there is no iOS-specific Expo step (the xcframework is bundled inside the pod, so `pod install` is enough).

### 4.5 Example app (`example/`)

A standard `react-native-builder-bob` example app. Demonstrates:

- `useFreeRasp(config, actions, raspExecutionStateActions)` mounted at `App.tsx`.
- Per-threat callbacks just flip an entry in a checks array → renders OK/NOK boxes.
- A modal flow for `storeExternalId` / `removeExternalId`.
- Buttons toggling `blockScreenCapture(true|false)` and reading `isScreenCaptureBlocked()`.
- An `addToWhitelist` loop over a hardcoded list at app start (Android-only).
- Manual location permission request (Android) — required because `locationSpoofing` detection needs `ACCESS_*_LOCATION` to evaluate spoofed providers.
- A custom `MalwareModal` that calls `getAppIcon(pkg)` lazily for each suspicious app.

The example is the *only* documentation of how the runtime location/permission requirements interact with detection — the README defers to a hosted GitBook, so consumers cloning the repo learn the gotchas only by reading this file.

## 5. Public API Surface (canonical)

```ts
// Hook (recommended entry-point)
useFreeRasp(
  config: TalsecConfig,
  actions: ThreatEventActions,
  raspExecutionStateActions?: RaspExecutionStateEventActions,
): void;

// Direct method calls
talsecStart(options: TalsecConfig): Promise<string>;
addToWhitelist(packageName: string): Promise<boolean>;     // Android-only
blockScreenCapture(enable: boolean): Promise<string>;
isScreenCaptureBlocked(): Promise<boolean>;
storeExternalId(data: string): Promise<string>;
removeExternalId(): Promise<string>;
getAppIcon(packageName: string): Promise<string>;          // Android-only, returns base64 PNG
onInvalidCallback(): void;                                 // kills app — internal use
removeListenerForEvent(channel: string): Promise<string>;  // internal use
```

`ThreatEventActions` enumerates **22 callbacks** (Android-superset). The hook silently ignores actions for which the platform has no native event — e.g. an iOS device will never call `actions.malware`.

## 6. Strengths, Weaknesses, Risks

### 6.1 Strengths

1. **Run-time-randomised channel/event identifiers** raise the bar for static analysis of the bundle. This is the single best architectural decision in the codebase.
2. **Cache-and-replay dispatcher** correctly handles the realistic case where the JS listener attaches *after* native checks have already fired (e.g. a slow JS bundle). Many naïve RN bridges drop those events.
3. **Foreground gating** on Android (via `LifecycleEventListener`) prevents background callback storms from waking the app or doing useless work.
4. **Lazy app-icon retrieval** keeps the malware payload small (Base64 PNGs would balloon a 50-app suspicious list to MBs). Sending icons on demand via `getAppIcon(pkg)` is the right trade-off.
5. **API 34/35 screen-capture detection** is surfaced cleanly with permission checks and helpful logging when a host forgets the manifest entries.
6. **One-hook ergonomics** (`useFreeRasp`) is genuinely friendly — most consumers will write `useFreeRasp(cfg, {root: () => kill(), …})` and be done.
7. **Expo plugin** removes the most painful step (manually adding the Talsec maven repo) — a non-trivial DX win in the Expo-managed-workflow world.
8. **Defence-in-depth** in nuclear callbacks: any deviation from the random ID protocol → `abort()`/`killProcess`, denying the attacker incremental probing.

### 6.2 Weaknesses

| # | Issue | Where | Severity |
|---|---|---|---|
| W1 | **No New-Architecture (Fabric/TurboModule) support.** Bridge-only. Future RN versions will deprecate the legacy bridge. | `package.json`, `freerasp-react-native.podspec`, `android/build.gradle` (codegen block exists but only when `newArchEnabled`) | High |
| W2 | **Singleton `isRaspStarted` + module-level `eventsListener`** make the SDK effectively a global. Multiple `useFreeRasp` calls in the same JS process race; second call's actions silently win or lose depending on timing. | `src/api/methods/reactNative.ts`, `src/api/listeners/threat.ts` | High |
| W3 | **`useEffect(..., [])` ignores config/actions changes.** If the host swaps action handlers (e.g. routes change), the *original* closure keeps firing — common React footgun. | `src/api/methods/reactNative.ts` line 50 — `eslint-disable-next-line react-hooks/exhaustive-deps` | High |
| W4 | **`NativeEvent` is loosely typed** (`{[k:string]: number\|string[]\|undefined}`). The 22-arm switch is order-coupled to `Threat.getValues()` order. Adding a threat is three-way coupled (ThreatEvent.kt, Threat.getValues(), threat.ts switch). | `src/types/types.ts`, `src/api/listeners/threat.ts`, `src/models/threat.ts` | Medium |
| W5 | **iOS RandomGenerator is non-CSPRNG** (`Int.random(in:)`). Asymmetric with Android's `SecureRandom`. | `ios/utils/RandomGenerator.swift` | Low–Medium |
| W6 | **iOS `getProtectedWindow` is multi-scene-blind.** First scene's first window only. Silent partial blocking on iPad multitasking, CarPlay, Mac Catalyst, Vision Pro. | `ios/utils/Utils.swift` | Medium |
| W7 | **Android `getAppIcon` PNG quality is 10**. That's *very* aggressive lossy compression for an icon — visually crunchy. | `android/.../utils/Utils.kt` `compressBitmap` | Low |
| W8 | **External ID storage on iOS is plain `UserDefaults`**, unencrypted, persists across reinstalls only if the app uses a backup-restore. No Keychain integration. Android delegates to `Talsec.storeExternalId`, which is opaque. | `ios/FreeraspReactNative.swift` `storeExternalId` | Medium |
| W9 | **No tests.** No `__tests__/` dir, no native unit tests. The example app *is* the test. | repo root | Medium |
| W10 | **Three-way coupling for new threats.** Outlined above — adding `automation` in 4.5.0 required edits in `ThreatEvent.kt`, `PluginThreatHandler.kt`, `Threat.getValues()`, `types.ts`, `listeners/threat.ts`, and matching iOS files when applicable. | repo-wide | Medium |
| W11 | **`Set`-semantics cache on Android** loses count of repeated threats — an attacker hooking-then-unhooking-then-hooking presents as a single hook event on resume. | `dispatchers/ThreatDispatcher.kt` | Low (acceptable for binary state) |
| W12 | **`onPause` on `isFinishing == true` calls `unregisterSDKListener`** but does not call `Talsec.stop` (the SDK has no public stop). The detection engine effectively stays running for the process lifetime. | `FreeraspReactNativeModule.kt` | Low |
| W13 | **No threat severity / no metadata** sent with the event. `actions.hooks()` knows *something* was detected but not *which* hook framework (Frida vs Xposed vs Shadow). | API design | Medium |
| W14 | **No per-threat backoff / dedupe window** in JS. The native side dedupes via `Set`, but if a threat re-fires across foreground/background cycles, the JS handler is called every cycle — acceptable but sometimes surprising. | dispatcher pattern | Low |
| W15 | **`isProd` defaults to `true`**. Forgetting to set this for development means false-positives kill the app in your dev environment. The README points to the GitBook for this nuance, which the repo doesn't ship. | `FreeraspReactNativeModule.kt` `getBooleanSafe("isProd")` | Low |
| W16 | **Expo plugin uses `buildGradle.includes(url)` for idempotency.** A user who replaces the URL with an internal mirror will get the original re-injected on every Expo prebuild. | `plugin/src/index.ts` | Low |
| W17 | **`FreeraspReactNativeModule.PluginListener` is created fresh on every `addListener` call**, but `ThreatDispatcher.listener` is a `lateinit var` — re-registration overwrites silently with no rebind notification. | `FreeraspReactNativeModule.addListener` | Low |
| W18 | **`itemsHaveType` is `every typeof === t`**, so a single null in the array passes (`typeof null === 'object'`, fails). However a string-typed test with empty array passes vacuously — `getThreatChannelData` could resolve `[]` and the check would pass (but length-mismatch catches that here). | `src/utils/utils.ts` | Low |
| W19 | **No obfuscation guidance for the JS bundle.** The package randomises native channel IDs but there's nothing the developer can do to hide the JS-side `Threat.PrivilegedAccess` constants (they're string-named in `models/threat.ts`). Hermes bytecode + Metro's identifier mangling is the only line of defence and it's weak. | repo-wide | Medium |
| W20 | **Two `useEffect` cleanups don't await each other** (`removeThreatListener`+`removeRaspExecutionStateEventListener` are fire-and-forget). On rapid mount/unmount cycles, late removals can race a fresh registration. | `src/api/methods/reactNative.ts` cleanup | Low |
| W21 | **No telemetry seam for host apps.** All events ultimately go through Talsec's report. There is no opinionated way to also forward to Sentry / Datadog / Crashlytics — every host re-implements this in their action handlers. | API design | Medium |

### 6.3 Risks

- **Vendor lock-in to Talsec's binary maven repo.** Any availability blip on `europe-west3-maven.pkg.dev/talsec-artifact-repository` blocks every CI build worldwide.
- **Closed-source detection engine** means the security posture is **as strong as Talsec's update cadence**. The 4.5.0 changelog mentions adding KernelSU + HMA support — if a new bypass tool drops mid-quarter, your protection lags.
- **Free tier "fair usage policy"** is not enforced in code; it's enforced administratively. A consumer overshooting could be silently degraded or rate-limited via a future binary update.
- **Self-protection by `abort()`** is hostile UX. A motivated attacker bypasses by replacing the function pointer; an honest user on a flaky device sees an unexplained crash.
- **No formal threat model published** for the package itself (only for the SDK). The protocol between native and JS — random IDs, base64-JSON malware payload — has no published assurance argument.

## 7. Operations & Lifecycle

### 7.1 Versioning

- The package follows SemVer. CHANGELOG entries are dual-keyed by SDK version (Android `TalsecSecurity-Community-ReactNative` + iOS `TalsecRuntime`). A patch release of either SDK floats up the RN package by a patch.
- React-native versions: peer-dependency on `react-native: *` — effectively claims compatibility with everything. Realistically tested against 0.72.x (RN devDep).

### 7.2 Distribution

- NPM tarball includes `lib/`, `src/` (for the `react-native` field), `android/`, `ios/` (with the **vendored xcframework**), and the Expo `plugin/build/` outputs.
- `.npmignore` keeps the `example/` and other non-shipping artefacts out.

### 7.3 CI

- `.github/workflows/ci.yml` runs lint, typecheck (root + plugin), prettier-check, then `yarn prepack`, then Android example build (with cached Gradle).
- No iOS build job visible in the snippet read.

## 8. Conclusion

`freerasp-react-native` is a **competent, opinionated bridge** that delivers a hard problem (mobile RASP) to React Native consumers with a single hook. Its standout idea is the per-process random channel/identifier negotiation between native and JS. Its main debts are New Architecture support, multi-instance JS hygiene, type-safety on the event payload, and the threefold-coupled threat-addition workflow. The next document, `02-superior-solution-proposal.md`, takes those concerns as the design forcing-function for a from-first-principles redesign.
